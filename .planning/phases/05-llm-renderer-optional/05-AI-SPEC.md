## 1. System Classification

**System Type:** Extraction (constrained structured output) — a single LLM call returns prose strings + per-meeting classification verdicts; explicitly NOT an agent (no tools, multi-turn, RAG, or orchestration).

**Description:**
An optional, swappable enhancement to the nightly "Evening Studio Check-in". On each weekday GitHub Actions cron run, ONE structured Anthropic call receives the already-computed, already-formatted facts of the studio report (it never sees or does arithmetic) and returns (a) on-brand prose for the Google Chat card — the header sentence and note/degraded wording — and (b) per-meeting keep/soften/drop verdicts that adjudicate the deterministic reconciler's "worth a look" calendar flags. Deterministic TypeScript assembles the final Cards v2 payload from these strings + verdicts; all numbers, rows, and figures stay deterministic. Users are the design studio's PMs/designers reading the card. "Good" = the card reads naturally and on-brand, the meeting flags are sensible, and — above all — every number is exact and the message always posts (falling back to the existing deterministic template on any LLM failure, with a visible degraded-mode note).

**Critical Failure Modes:**
<!-- The behaviors that absolutely cannot go wrong in this system -->
1. The LLM alters, fabricates, or re-computes ANY number/hour/figure (it must only restate pre-computed facts) — this destroys trust and is the project's cardinal sin.
2. An LLM failure (timeout, bad/missing key, malformed or schema-invalid output, guardrail trip) causes a missed or broken post instead of a clean fallback to the deterministic template + degraded note (violates REL-01 "never silently skip a night").
3. Fuzzy meeting judgment produces false confidence — dropping a flag that should have stood, or inventing/over-asserting a flag the reconciler didn't raise (the reconciler biases to silence; the LLM must not undo that bias recklessly).
4. The LLM injects unsafe/unescaped content into the Cards v2 HTML, or drifts off-brand into hype/filler.
5. Unbounded cost/latency on the nightly call (must stay a few cents/night, single cached call).

## 1b. Domain Context

> Researched by `gsd-domain-researcher`. Grounds the evaluation strategy in domain expert knowledge.

**Industry Vertical:** Creative/marketing agency operations — internal team tooling (PSA-adjacent: resourcing/capacity nudges over Productive.io + Google Calendar). NOT a regulated vertical.

**User Population:** A handful of studio project managers plus three monitored designers (and informally the creative lead / studio manager), reading a single nightly Google Chat card. Non-technical, time-pressed, often reading on the way out the door at ~4:30pm. They are NOT evaluating an AI — they are deciding, in seconds, "is this worth acting on before tomorrow?"

**Stakes Level:** Low-to-Medium operationally — a wrong nudge wastes a little attention, not safety or money. BUT trust is the entire product: the moment a number is wrong, a flag is unfair, or the tone reads robotic/hype, the team stops reading and the tool silently dies. So the *effective* stakes on tone, flag fairness, and numeric fidelity are High even though the operational blast radius is small.

**Output Consequence:** A PM reads the card and either fixes something before the next working day (writes a missing brief, fills an underbooked designer's day, books an unaccounted meeting) or registers "all sorted" and moves on. There is no automated downstream action — every consequence is a human deciding whether to trust and act. The failure that matters is not a bad action; it is *disengagement* — one robotic or unfair message and people quietly stop reading.

### Known Failure Modes in This Domain

<!-- How failure actually manifests HERE, not generic hallucination. -->

1. **Tone drift into AI/corporate slop.** The prose slides to the "internet-average" voice — "I hope this message finds you well", "let's dive in", forced triads, gratuitous em-dashes, hype positivity. It still parses and posts, so it fails silently against *adoption*: the team starts skimming, then stops reading.
2. **Over-flagging meetings that obviously didn't need a booking.** The LLM "helpfully" keeps or hardens a borderline calendar item the deterministic reconciler only tentatively raised. A few unfair flags and PMs learn the flags aren't credible — classic alert fatigue, and the most likely way this specific tool loses trust.
3. **Blame-y framing.** Wording that makes a designer feel audited or implies a named PM dropped the ball. Adoption depends on the message feeling like collective help; this inverts it.
4. **Numeric contradiction (cardinal sin).** The prose restates, rounds, or infers an hour figure that disagrees with the deterministic rows beside it. Even one visible mismatch breaks the whole "trustworthy heads-up" premise and the team stops believing any number in the card.

One design nicety (not a regulation): avoid surfacing potentially sensitive meeting *titles* verbatim in the public Chat card where it can be avoided — prefer neutral phrasing for an unaccounted meeting. This is courtesy/tone, not a legal constraint.

## 2. Framework Decision

**Selected Framework:** Official Anthropic TypeScript SDK (`@anthropic-ai/sdk`) used directly — single Messages API call, structured JSON output validated with the project's existing zod at the boundary, prompt caching on the stable on-brand system prompt. No agent framework.

**Version:** `@anthropic-ai/sdk` latest `^0.x` (pin at install). Stack: Node 22, TypeScript ~5.9 (ESM), zod ^4, native `node:test`, tsx ^4.

**Rationale:**
The Phase 5 workload is exactly ONE structured LLM call per nightly cron — prose strings for the Cards v2 message plus per-meeting keep/soften/drop verdicts — with no tools, multi-turn, RAG, or orchestration. Any agent framework is dead weight and the documented anti-pattern ("direct SDK call is less code"). The provider is a fixed sanctioned `ANTHROPIC_API_KEY`, and the stack is already TypeScript + zod + node:test, so the first-party SDK drops cleanly behind the existing `RenderMessage` signature with a try/catch fallback to `renderTemplate` — satisfying the never-skip-a-night rule with the least abstraction and the smallest dependency surface. Note: prefer a standard Messages call returning a JSON object that is then parsed and **zod-validated at the boundary** (zod is the single source of truth) over trusting the model's schema adherence alone.

**Alternatives Considered:**

| Framework | Ruled Out Because |
|-----------|------------------|
| Vercel AI SDK (`ai` + `@ai-sdk/anthropic`, `generateObject` + zod) | Adds a provider-agnostic abstraction layer this Anthropic-pinned single call doesn't need today. Reconsider only if multi-provider support becomes a real requirement. |
| Claude Agent SDK | Built for tool-use/MCP/code agents — wrong shape and heavier than one structured call needs. |
| LangGraph / LangChain / CrewAI / LlamaIndex | Orchestration/RAG abstractions for multi-step agents; overkill for a single structured extraction call, and several fail the TypeScript/Anthropic-only fit. |

**Vendor Lock-In Accepted:** Yes (intentional) — Anthropic-only is a hard project constraint (sanctioned org `ANTHROPIC_API_KEY`). The swappable `RenderMessage` interface + deterministic fallback keep the blast radius contained; the Vercel AI SDK is the documented exit if provider-agnosticism is ever needed.

### Entry Point Pattern
The whole Phase-5 surface is ONE `messages.create` call per nightly run: a cached system prompt
(constant on-brand voice + rules) + a user message of pre-computed facts → a JSON string →
`JSON.parse` → **zod parse** → typed result. The model produces only prose + verdicts; it never
sees or emits a number it computed.

```ts
// src/llm/client.ts — thin wrapper. One client, per-request timeout, bounded retries.
const anthropic = new Anthropic({
  apiKey: process.env.ANTHROPIC_API_KEY, // sanctioned org key; throws if unset at call time
  maxRetries: 2,                          // SDK default; retries 429/5xx/network with backoff
  timeout: 20_000,                        // hard ceiling — a nightly job must never hang
});

// The model's contract: prose strings + one verdict per worth-a-look meeting. zod is truth.
const LlmOutput = z.object({
  headerSentence: z.string().min(1).max(200),
  // verdicts keyed by a stable index we put in the facts payload (NOT by free-text title).
  meetingVerdicts: z.array(
    z.object({
      id: z.number().int(),                       // echoes the id we sent in
      verdict: z.enum(["keep", "soften", "drop"]),
      line: z.string().max(160),                  // the rendered 📅 sub-line text (no numbers)
    }),
  ),
});
type LlmOutput = z.infer<typeof LlmOutput>;

const msg = await anthropic.messages.create({
  model: "claude-haiku-4-5-20251001",   // tier decision in §4; Haiku is the default
  max_tokens: 1024,                      // ALWAYS set — bounds cost + truncation (see pitfalls)
  system: [
    {
      type: "text",
      text: SYSTEM_PROMPT,               // constant on-brand voice + rules + JSON contract
      cache_control: { type: "ephemeral" }, // cache the stable prefix — it never changes nightly
    },
  ],
  messages: [
    { role: "user", content: JSON.stringify(facts) }, // pre-computed, pre-rounded facts only
    { role: "assistant", content: "{" },              // prefill: force the reply to start as JSON
  ],
});

// Extract: a non-streaming Message has content blocks; the first is our text block.
if (msg.stop_reason === "max_tokens") throw new Error("truncated — raise max_tokens or fall back");
const block = msg.content[0];
const raw = block.type === "text" ? "{" + block.text : ""; // re-attach the prefilled "{"
const parsed = LlmOutput.parse(JSON.parse(raw));           // zod throws → caller falls back
```

```ts
// src/llm/renderLlm.ts — satisfies the SAME RenderMessage-style contract, wrapped in fallback.
export async function renderLlmOrTemplate(
  report: StudioReport,
  ctx: RenderContext,
): Promise<ReturnType<RenderMessage>> {
  try {
    const out = await callModel(report, ctx);   // the create()+zod parse above
    return assembleCardsV2(report, ctx, out);   // DETERMINISTIC assembly: numbers stay in TS
  } catch (err) {
    console.warn(`LLM render failed, using template: ${String(err)}`); // visible, never silent
    return renderTemplate(report, withDegradedNote(ctx)); // existing renderer + degraded note
  }
}
```

### Key Abstractions
| Concept | What It Is | When You Use It |
|---------|-----------|-----------------|
| `Anthropic` client | The SDK entry object; holds `apiKey`, `timeout`, `maxRetries`. Construct ONCE per process. | Module-level singleton in `src/llm/client.ts`. |
| `messages.create()` | The single Messages API call. Non-streaming returns a `Message`; pass `{ stream: true }` for events. | The one call per nightly run. Use non-streaming (we need the whole JSON before validating). |
| `system` as `TextBlockParam[]` | System prompt as an array of blocks so a block can carry `cache_control`. | Put the constant on-brand voice + JSON contract in a cached block. |
| `cache_control: { type: "ephemeral" }` | Prompt-caching breakpoint. 5-min TTL by default; cache reads cost 0.1× input. | On the stable system block — it is byte-identical every night. |
| `RequestOptions` (2nd arg) | Per-call `{ timeout, maxRetries, signal }` overriding client defaults. | Tighten/loosen the ceiling for this one call without a new client. |
| Response `content[]` + `stop_reason` | Message body is an array of blocks; `content[0].text` is the text. `stop_reason === "max_tokens"` means truncated. | Extract text; treat truncation/`refusal` as a hard failure → fallback. |
| zod schema (`.parse`) | The project's existing validator — the REAL output contract. | Validate every field at the boundary; on throw, fall back to the template. |

### Common Pitfalls
1. **Trusting the model to emit valid JSON.** Even with a JSON instruction the model can wrap output in prose or a ```` ```json ```` fence, or trail a comma. Mitigations used here: an **assistant prefill of `"{"`** (forces the reply to open as a bare object), then `JSON.parse` inside the `try`, then **zod** — any failure throws into the fallback. Do NOT hand-roll regex to "fix" the JSON.
2. **Leaving `max_tokens` high or unset.** It is required by the API, but a loose value invites runaway cost and lets a chatty model overrun. Set it tight (~1024 here) AND check `stop_reason === "max_tokens"` — a truncated body parses as broken JSON and must fall back, never post a half-card.
3. **Setting `temperature` on Sonnet 4.6.** Temperature is **deprecated for models released after Opus 4.6 (which includes Sonnet 4.6)** — any value other than `1.0` returns a 400. Omit `temperature` entirely; it is safe on Haiku 4.5 but omit it everywhere for forward-compat. (Verified: SDK `MessageCreateParamsBase` deprecation note.)
4. **No timeout → a hung nightly job.** The default client retries but the run is unattended on cron; a stalled socket would silently miss the night. Always pass `timeout` (client or per-request) so a slow call fails fast into the deterministic fallback (REL-01).
5. **Letting the model near the numbers.** The facts payload must contain only **pre-rounded display strings** (e.g. `"3.5h"`), never raw minutes, and the assembled card must take every figure from `report`/`ctx` in TS — never from the model's `line`. Strip/escape any model text before it enters Cards v2 HTML (the renderer already has `escapeHtml`). This is the project's cardinal trust rule.
6. **Per-call client construction.** `new Anthropic()` per run is fine for one nightly call but wasteful in tests; construct once and inject via the existing `RunNightlyDeps` seam so `node:test` can stub it with zero network.

## 4. Implementation Guidance

**Model Configuration:**

| Setting | Value | Why |
|---------|-------|-----|
| `model` (default) | `claude-haiku-4-5-20251001` (Haiku 4.5) | The task is restating ~10 short facts in a fixed voice + classifying ≤9 meeting titles. This is well within Haiku's range and keeps cost at well under 1¢/night. Start here. |
| `model` (upgrade path) | `claude-sonnet-4-6` (Sonnet 4.6) | Switch ONLY if eval (§5) shows Haiku drifting off-brand or mis-judging meetings. Sonnet is ~3× input / 3× output cost — still a fraction of a cent/night — so the cost penalty is trivial; the reason to default to Haiku is latency + "smallest tool for the job", not money. |
| `max_tokens` | `1024` | Output is one short sentence + a handful of verdict lines (~150–300 tokens typical). 1024 leaves headroom while bounding cost/truncation. Check `stop_reason` regardless. |
| `temperature` | **omit** | Deprecated on Sonnet 4.6 (400 on any non-1.0 value). Omitting gives stable, low-variance prose — ideal for an on-brand restatement task. Do not set it on Haiku either, for forward-compat. |
| `system` | cached `TextBlockParam[]` with `cache_control: { type: "ephemeral" }` | The voice + rules + JSON contract are byte-identical every night → cache read at 0.1× input after the first call within the 5-min window (negligible here since runs are daily, but free to leave on). |
| `timeout` | `20_000` ms (client-level) | Unattended cron must fail fast into the template, never hang. |
| `maxRetries` | `2` (SDK default) | Auto-retries transient 429/5xx/network with backoff; exhaustion throws → fallback. |

Model IDs are pinned/dated by intent so a silent model swap can't change the card's voice unnoticed; bump deliberately and re-run the eval set.

**Core Pattern:**
Single-shot constrained extraction with deterministic assembly and a total fallback. Sequence:

1. **`buildFacts(report, ctx)`** (pure, in `src/llm/prompt.ts`) → a small JSON object of *display-only* facts: the busy/clean verdict flag, each designer's pre-rounded `*Hours` strings, brief-flag presence, and the worth-a-look meetings as `{ id, title, durationLabel }` (id = stable array index; NO minutes, NO capacity numbers). This is the user message.
2. **`messages.create`** with the cached system prompt + the facts user message + an assistant prefill of `"{"`.
3. **Parse + validate**: re-attach `"{"`, `JSON.parse`, then `LlmOutput.parse` (zod). Any throw → caught by the caller.
4. **`assembleCardsV2(report, ctx, out)`** (pure, deterministic) builds the `CardsV2Payload` using `out.headerSentence` and, per meeting, `out.meetingVerdicts` to decide keep/soften/drop — but every NUMBER, deep-link, week-bar dot, and designer row still comes from `report`/`ctx` exactly as `renderTemplate` does today. The model's `line` text is escaped via `escapeHtml` before insertion.
5. **Fallback**: the whole thing is wrapped in `renderLlmOrTemplate`'s try/catch → `renderTemplate(report, withDegradedNote(ctx))` on any failure, with a visible muted note (mirrors the existing `calendarUnavailable` note pattern). Never a missed post (REL-01).

The meeting-verdict semantics adjudicate the reconciler's bias-to-silence (D-04): `keep` = surface the 📅 line as the reconciler raised it; `soften` = surface with gentler wording; `drop` = suppress this flag. Because the reconciler already biases hard to silence, the assembler should treat an unexpected/unknown `id` or a `drop` of an item the reconciler didn't raise as a no-op — the model can only soften or remove, never invent a flag.

**Tool Use:**
None. This is deliberately a plain Messages call — no `tools`, no `tool_choice`, no function calling. Structured output is achieved by JSON-prompt + assistant prefill + zod validation, which is simpler and cheaper than tool-call JSON for a single fixed schema and keeps the trust boundary (zod) explicit. (Anthropic's tool/JSON-mode would also work, but adds tool-system-prompt tokens and a second place schema lives; zod stays the single source of truth.)

**State Management:**
Stateless. There is no conversation, no memory, no persistence — one isolated call per nightly run. The only "state" is the cached system prompt (a 5-min provider-side cache, irrelevant across daily runs) and the injected `RunNightlyDeps` for testing. No DB, no vector store, no session.

**Context Window Strategy:**
Non-issue by construction. Input is one tiny facts object (three designers, ≤~9 meetings) — on the order of a few hundred tokens plus the cached system prompt. We never approach the window, so no chunking/truncation/summarisation is needed. The discipline is the inverse: keep the facts payload **minimal and number-free**, not "fit a large context". `max_tokens` (1024) bounds the output side.

## 4b. AI Systems Best Practices

> Written by `gsd-ai-researcher`. Cross-cutting patterns every developer building AI systems needs — independent of framework choice.

### Structured Outputs with zod (TypeScript — this project's equivalent of "Pydantic")

This is a TypeScript project; **zod ^4** plays the role Pydantic plays in Python and is already a
dependency. zod is the **single source of truth** for the output contract — we do NOT rely on the
model's schema adherence. The model is told the shape in the prompt; zod *enforces* it at the
boundary and any deviation throws into the deterministic fallback.

```ts
// src/llm/schema.ts — the real output contract for this Extraction use case.
import { z } from "zod";

export const LlmOutput = z.object({
  // (a) on-brand prose for the card — bounded length, no numbers (numbers come from TS).
  headerSentence: z.string().min(1).max(200),
  // (b) one verdict per worth-a-look meeting, keyed by the id we sent in (not by title).
  meetingVerdicts: z
    .array(
      z.object({
        id: z.number().int().nonnegative(),
        verdict: z.enum(["keep", "soften", "drop"]),
        line: z.string().max(160), // rendered 📅 sub-line; escaped before Cards v2 insertion
      }),
    )
    .max(20), // hard cap — a runaway list is a failure, not a card
});
export type LlmOutput = z.infer<typeof LlmOutput>;
```

**How the SDK integrates:** the SDK has no built-in zod coupling for plain Messages calls (that's a
feature of the *beta* tool helpers). We use the explicit, auditable path: JSON prompt + assistant
prefill `"{"` → `JSON.parse` → `LlmOutput.parse`. This keeps validation in ONE obvious place.
(If a future need arises, `@ai-sdk/anthropic`'s `generateObject(schema)` or the SDK's beta
`betaZodTool` would push schema enforcement into the call — but they hide the boundary; for a
trust-critical post we keep zod explicit.)

**Retry logic:** the SDK already retries transport failures (`maxRetries: 2`, 429/5xx/network).
We do **NOT** add an LLM-content retry loop ("ask it again to fix the JSON") — for a once-a-night
job that re-spend isn't worth it and a second bad parse just delays the inevitable. Policy: **one
call, one validation; on ANY failure (transport-exhausted, JSON parse, zod parse, `stop_reason ===
"max_tokens"`/`"refusal"`) log a one-line `console.warn` with the failure class (never the key,
never the webhook URL) and fall back to `renderTemplate` + a visible degraded note.** Surface =
the degraded note in the card itself, so a persistently-broken LLM is noticed by its absence of
on-brand prose, not by silence.

### Async-First Design

`messages.create()` returns a Promise (an `APIPromise<Message>`); the entire renderer is `async`
and is `await`-ed inside `runNightly`, which is already async. There is no event loop of our own —
GitHub Actions runs `node --import tsx src/index.ts` to completion and exits.

- **The one common mistake to avoid:** do NOT call a top-level `asyncio`-style blocking wrapper or
  spin a second runtime — `runNightly` is already the awaited async root (`src/index.ts` line 265).
  Just `await renderLlmOrTemplate(...)` where `renderTemplate(...)` is called today. Also do not
  fire the call without `await` (a floating promise would let the process exit before the post).
- **Stream vs. await:** **await, do not stream.** Streaming is for incremental UX; here we need the
  *whole* JSON object before we can `JSON.parse` + zod-validate it, and there is no human watching a
  cursor at 4:30pm. Non-streaming `messages.create` returns a single `Message` — simpler, and the
  only correct choice when the output must be validated atomically before use.
- **Timeout is the async safety net:** the client `timeout: 20_000` (or a per-call `RequestOptions`)
  guarantees the awaited promise rejects rather than hanging the unattended job.

### Prompt Engineering Discipline

- **System vs. user separation (load-bearing here):** the **system** prompt holds everything
  constant — the Solvd on-brand voice (warm, plain, a soft collective nudge; never hype, never
  "conflict"; D-04), the hard rules ("you receive pre-computed facts; restate them; NEVER compute,
  infer, or alter a number"), the meeting keep/soften/drop semantics, and the exact JSON contract.
  The **user** message holds ONLY this run's facts object. This split is what makes prompt caching
  work (the system block is byte-identical nightly) and keeps the variable surface tiny.
- **Few-shot:** **inline, in the system prompt**, not dynamic retrieval (there's nothing to retrieve
  from). Two or three short examples — a busy night, a clean night, an ambiguous meeting that should
  be `soften`ed — anchor the voice and the verdict bias far better than instructions alone. Keep
  them in the cached system block so they cost cache-read rates, not fresh input every night.
- **`max_tokens` is mandatory and tight:** `1024`. Never leave output unbounded in production — it
  is the single most effective guard against both runaway cost and a chatty model overrunning the
  schema. Pair it with the `stop_reason === "max_tokens"` check (a truncated body is a hard failure).
- **Number-free facts:** the prompt's strongest trust lever is simply not giving the model anything
  it could miscompute — send display strings (`"3.5h"`), never raw minutes or capacity figures.

### Context Window Management

This is an **Extraction** system with a deliberately tiny, fixed input (three designers, ≤~9
meetings) — it will never approach the context window, so there is no RAG chunking, no conversation
summarisation, and no agent compaction to manage. The relevant discipline is the opposite of "fit
more in":

- Keep `buildFacts` **minimal** — only what the model must restate or judge; omit everything the
  deterministic assembler already owns (rows, dot counts, deep links).
- Bound the **output** side with `max_tokens` and the zod `.max()` array cap, so a degenerate
  response can't balloon.
- The cached system prompt is the only large, stable block; everything else is small and per-run.
  If the system prompt ever grows (more few-shot, longer voice guide), it stays in the single
  cached block so its cost is amortised, not the variable user payload.

### Cost and Latency Budget

Volume: **one call per weekday ≈ 22 calls/month.** Sizing assumption (verify with the real prompt
via `/v1/messages/count_tokens`): cached system prompt ~1,500 tokens, per-run facts ~400 tokens,
output ~300 tokens.

Current first-party prices (platform.claude.com pricing, current as of this spec):

| | Haiku 4.5 | Sonnet 4.6 |
|---|---|---|
| Input | $1 / MTok | $3 / MTok |
| Output | $5 / MTok | $15 / MTok |
| 5-min cache write | $1.25 / MTok | $3.75 / MTok |
| Cache read | $0.10 / MTok | $0.30 / MTok |

Per-night estimate (system prompt billed as a fresh cache *write* each night, since daily runs are
> 5 min apart, so no cross-run cache hit — the cache_control is effectively free insurance, not a
saving here):

- **Haiku 4.5:** ~1,500 × $1.25/M (cache write) + ~400 × $1/M (facts) + ~300 × $5/M (output)
  ≈ **$0.0019 + $0.0004 + $0.0015 ≈ ~$0.004/night ≈ ~$0.08/month.**
- **Sonnet 4.6:** ~1,500 × $3.75/M + ~400 × $3/M + ~300 × $15/M
  ≈ **$0.0056 + $0.0012 + $0.0045 ≈ ~$0.011/night ≈ ~$0.25/month.**

Both are comfortably inside the "few cents/night" constraint; Sonnet's premium is immaterial in
absolute terms, so the model choice is a quality/latency call, not a budget one. (Estimates are
order-of-magnitude — token counts are assumptions; confirm against the finished prompt. Pricing is
current but can change.)

**Latency:** a single small Haiku call is typically ~1–3s; Sonnet ~2–5s. Well inside the 20s
timeout, and irrelevant to a once-nightly unattended job — there is no user waiting.

**Caching:** prompt caching on the system block is enabled (`cache_control: ephemeral`) as free
insurance, but provides no real saving at one-run-per-day cadence (5-min TTL). There is **no
semantic cache and no exact-match cache** — every night's facts differ and a stale cached card
would be actively harmful (the whole point is *tonight's* numbers). Do not cache the output.

**Cheaper-model routing:** already applied — **Haiku is the default**, the cheap model, doing the
only sub-task there is. There are no sub-tasks to route (no separate classification/routing/
summarisation step); the single call does prose + verdicts together. Escalate the whole call to
Sonnet only if eval demands it.

## 5. Evaluation Strategy

> Right-sized to a SMALL internal nightly tool: one structured Anthropic call per weekday,
> TypeScript / Node 22, native `node:test`, zod ^4. The eval rigor is deliberately lopsided —
> the bulk of the assurance is **deterministic code-based property tests** over golden fixtures
> (the pattern this repo already uses in `src/render/__tests__/`), NOT a metrics dashboard.
> The single most important eval is a STRUCTURAL invariant, not a graded score: prove the LLM
> can never alter a number. Tone is a cheap offline judge run during dev. A heavy Python
> observability stack (Arize Phoenix / DeepEval / RAGAS) is explicitly OUT OF SCOPE and off-stack.

### Dimensions

The two Critical dimensions are enforced as **hard invariants** (property tests + a runtime
guardrail), not as graded metrics — a single failure is a fallback, not a low score. The High
dimensions are quality signals run offline during dev/eval.

| Dimension | Rubric (Pass / Fail) | Measurement Approach | Priority |
|-----------|---------------------|----------------------|----------|
| **Number fidelity (the cardinal invariant)** | PASS: for ANY LlmOutput — including adversarial/garbage prose that states fake hours — every numeric figure, designer row, week-bar dot count, and deep-link in the assembled `CardsV2Payload` is byte-identical to what `renderTemplate(report, ctx)` produces from the same `report`/`ctx`. The LLM's prose contributes the header sentence and per-meeting lines ONLY — never a figure. FAIL: any number in the assembled card traces to the model's output rather than to `report`/`ctx`. | **Code** — `node:test` property test: feed `assembleCardsV2` a battery of hostile `LlmOutput` objects ("Sarah has 99 hours free", injected `<b>0h</b>`, numbers in `headerSentence`/`line`) and assert every numeric/row/dot/link field equals the `renderTemplate` baseline. This is the most important test in the phase. | **Critical** |
| **Fallback integrity (never skip a night — REL-01)** | PASS: on EVERY failure class — thrown SDK error, network timeout, `stop_reason==="max_tokens"`/`"refusal"`, non-JSON body, JSON that fails `LlmOutput.parse`, an empty/garbage response — `renderLlmOrTemplate` returns a complete, postable payload from `renderTemplate(report, withDegradedNote(ctx))` and logs one `console.warn` with the failure class. FAIL: any failure class throws out of the renderer, returns a partial card, or posts silently without the degraded note. | **Code** — `node:test` table-driven: a stubbed Anthropic client (injected via `RunNightlyDeps`) returns each failure class in turn; assert a valid payload + the degraded note + exactly one warn line, and that no secret/key/webhook text appears in the log. | **Critical** |
| **Flag fairness (kept flags are ones a PM would agree with)** | PASS: the LLM NEVER `drop`s a flag the reference set labels as genuine client work the reconciler rightly raised (recall-protective on real flags); softening a borderline one is acceptable; it never hardens or invents a flag (unknown/invalid `id`, or a `drop` of a flag the reconciler didn't raise, is a no-op by construction). FAIL: drops/suppresses a labelled-genuine flag, or asserts a flag the reconciler did not raise. Bias to silence (per §1b) — softening is cheap, dropping a real one is the costly error. | **Code** (structural: unknown-id / invent-a-flag is a no-op — assert in `node:test`) + **LLM Judge / Human** (the keep/soften/drop call itself, scored against the PM-labelled reference set below; precision-focused — a false flag costs more than a miss). | **High** |
| **On-brand voice (warm, plain, a collective nudge — not AI slop)** | PASS: reads like a colleague's quick heads-up — warm, plain, concise; frames things as "worth sorting", sounds like Solvd. FAIL: AI-slop tells — "I hope this finds you", "let's dive in", forced rule-of-three, em-dash overuse, hype adjectives, hollow positivity, generic-SaaS flatness. | **Code** (cheap regex/lint for the known slop tells — string-match a deny-list, count em-dashes, flag triads) as a fast first gate, then **LLM Judge** calibrated against the creative lead's human "sounds like us" scores. Dev/eval only — NOT in the nightly path. | **High** |
| **Non-blame framing (helps the team, never points a finger)** | PASS: references thin jobs / open hours by job and by designer-with-open-time; reads as collective "let's get tomorrow ready". FAIL: names or implies a PM at fault, or frames a designer as audited ("Sarah hasn't booked her time" vs "there's open time on Sarah's day"). | **LLM Judge** (rubric check on the header sentence + lines) with **Human** spot-review; backstopped by the structural fact that PM names never enter the facts payload. | **High** |
| **Brevity & scannability (length scales with severity — MSG-05)** | PASS: terse on a clean night, only as long as the night warrants; one-line verdict graspable at a glance. FAIL: padding/preamble, restating the card's own rows in prose, a wall of text on a quiet night. | **Code** — `headerSentence`/`line` length caps already enforced by the zod `.max()` bounds (`headerSentence ≤ 200`, `line ≤ 160`); an eval-time check asserts a clean-night header is materially shorter than a busy-night one. | **Medium** |

### Eval Tooling

**Primary tool: native `node:test` + committed golden/labelled fixtures.** This phase adds NO new
eval dependency. The deterministic invariants (number fidelity, fallback integrity) and the
structural half of flag fairness are property tests in `src/llm/*.test.ts`, mirroring the existing
`src/render/__tests__/renderMessage.test.ts` golden-fixture pattern (`assert.deepStrictEqual`
against committed expected JSON, stubbed dependencies via `RunNightlyDeps` — zero network).

The subjective dimensions (the keep/soften/drop call, on-brand voice, non-blame framing) are a
small **offline judge harness**: a one-off `tsx` script that runs the real prompt over the
labelled reference set and prints a pass/fail table — run by Liam during dev and when the prompt
or model ID changes, NOT on every CI run and NOT in the nightly path. The voice slop-lint is a
plain deny-list regex (reuse the project's humanizer-style slop list if present).

**Setup:**
```bash
# No new install. The eval stack is already present:
#   node:test (built-in), tsx ^4, zod ^4. The Anthropic SDK is stubbed in unit tests.
# Deterministic invariant + structural tests (no network, no key needed):
node --import tsx --test "src/llm/**/*.test.ts"

# Offline subjective harness (calls the real model over the labelled set — needs a dev key).
# Uses the personal $5-credit ANTHROPIC_API_KEY (LLM-02), never the prod org key:
ANTHROPIC_API_KEY=$DEV_KEY tsx scripts/eval-llm-renderer.ts
```

**CI/CD Integration:**
```bash
# CI runs ONLY the deterministic, network-free tests — the trust-critical invariants gate merges.
# (Same runner the repo already uses; the LLM client is stubbed, so no key in CI, no flake, no cost.)
node --import tsx --test "src/**/*.test.ts"
# The subjective harness is intentionally NOT in CI (it costs cents + needs calibration); it is a
# manual gate before changing the prompt, the model ID, or the few-shot examples.
```

### Reference Dataset

**Size:** ~25–30 labelled meeting events (already exists) for flag fairness, plus ~6 hand-built
report/context scenarios for voice + brevity (clean night, lightly-busy, fully-busy, calendar-only
failure, all-on-leave, holiday) — comfortably above the 10–20 floor without manufacturing filler.

**Composition:**
- **Flag fairness — reuse the Phase-4 labelled data, do not relabel from scratch:**
  - `src/calendar/__fixtures__/labelled-events.json` — ~4 weeks of REAL meetings already tagged
    `counts/…` (covered vs worth-a-look), `not-work`, overhead, solo, declined, etc., with golden
    Cases A (covered → must stay silent) and B (worth-a-look → must flag) pinned.
  - `.planning/phases/04-calendar-meeting-reconciliation/spike-output.{md,json}` (gitignored) —
    the human-labelled spike output over the same real calendar window.
  - Critical paths: covered meeting the LLM must not surface; genuine worth-a-look it must never
    `drop`. Edge cases: ambiguous/double-match titles (reconciler already returns null → nothing
    to judge); borderline internal-ish meetings (acceptable to `soften`). Adversarial: an `id` the
    reconciler never raised, and an out-of-range `id` (must be a no-op).
- **Voice + brevity + non-blame:** ~6 deterministic `StudioReport` + `RenderContext` scenarios
  reused/adapted from `src/render/__tests__/fixtures/` so the inputs are already trusted; the judge
  scores only the model's prose, never the numbers.
- **Number fidelity + fallback:** synthetic adversarial `LlmOutput` objects authored in the test
  file (fake numbers in prose, HTML injection, truncated/garbage bodies, each failure class) — these
  are code, not data, and live beside the property test.

**Labeling:**
- **Flag fairness ground truth → Studio PM / resourcing owner** (the keep/soften/drop a PM would
  actually make). Liam has already labelled ~4 weeks of real meetings; the PM is the arbiter for
  any new/borderline cases. The rule the eval enforces: **the LLM must never DROP a flag labelled as
  genuine client work; softening borderline ones is acceptable.**
- **On-brand voice calibration → Creative lead / studio manager** (the "sounds like us" arbiter);
  the LLM judge must be calibrated against their human scores (target ≥ 0.7 agreement) before its
  voice verdicts are trusted (§1b expert roles; ai-evals calibration guidance).
- **Number-fidelity & non-blame invariants → Liam** (builder) as code-checkable rules.

## 6. Guardrails

> Keep guardrails minimal and CHEAP — they sit in the nightly path. All online guardrails here are
> deterministic TypeScript (zod, string assembly, escaping); there is no second LLM "judge in the
> loop" at runtime. Every guardrail's failure action is the SAME: fall back to `renderTemplate` +
> degraded note (REL-01), never a missed or broken post.

### Online (Real-Time)

| Guardrail | Trigger | Intervention |
|-----------|---------|--------------|
| **zod boundary validation** | Model output fails `JSON.parse` or `LlmOutput.parse` (wrong shape, missing field, bad enum, array over `.max(20)`). | **Fall back** to `renderTemplate(report, withDegradedNote(ctx))`; one `console.warn` with the failure class. |
| **Number-invariance assembly (the cardinal guardrail)** | Always on — structural, not a check. `assembleCardsV2` takes every figure/row/dot/link from `report`/`ctx`; the model's `headerSentence`/`line` are inserted as PROSE only and are never parsed for numbers. | **Block by construction** — there is no code path by which a model number reaches a figure field. The property test (§5) proves it; this guardrail is the design that makes the test pass. |
| **HTML escaping of all model prose** | Any model-produced string (`headerSentence`, each `line`) before it enters the Cards v2 HTML-subset text. | **Sanitise** via the existing `escapeHtml` (already used in `rows.ts`/`renderMessage.ts`); prevents tag/script/markup injection into the card. |
| **Length / voice sanity cap** | `headerSentence > 200` chars or any `line > 160` chars (enforced by zod `.max()`), or output truncated (`stop_reason === "max_tokens"`). | **Fall back** to the template — a runaway or truncated body is treated as a failure, never posted as a half-card. |
| **Invent / harden-a-flag suppression** | Model returns a verdict for an `id` the reconciler never raised, or `drop`s an item not in the worth-a-look set. | **No-op** — the assembler ignores unknown ids and can only `soften`/remove a real flag, never add one (matches the reconciler's bias-to-silence, D-04). |
| **Timeout ceiling** | The call exceeds the 20 s client/per-request timeout. | **Fall back** — the awaited promise rejects fast into the template; the unattended job never hangs. |
| **Fallback-and-degraded-note contract (REL-01)** | ANY of the above trips, or any other thrown error in the renderer. | **Always post** the template card with a visible muted degraded note, so a dead LLM is noticed by the absence of on-brand prose, never by silence. |

### Offline (Flywheel)

| Metric | Sampling Strategy | Action on Degradation |
|--------|------------------|----------------------|
| **Fallback rate** (nights on template vs LLM) | Read from the nightly run-log line (§7) across recent runs. | A sustained rise = the LLM path is quietly failing (stale/blocked key, schema drift, model change). Investigate the logged failure class; fix or accept template-only until fixed. |
| **Flag-fairness drift** | When the PM disagrees with a kept/softened/dropped flag on a real card, add that meeting to `labelled-events.json` and re-run the offline judge harness. | If the LLM `drop`ped a genuine flag, treat as a Critical-priority prompt/few-shot fix and re-run the flag-fairness eval before shipping the prompt change. |
| **Voice drift toward slop** | Periodic creative-lead review of recent header sentences + the slop-lint deny-list hit rate. | Tune the voice rubric + few-shot examples in the cached system prompt; re-calibrate the LLM judge against fresh human scores. |
| **Per-night cost / token trend** | The run-log's input/output token + est-cost fields, eyeballed across the month. | Should stay sub-cent on Haiku; an unexplained climb (longer prompt, model swap, retry storm) prompts a prompt-size / model-ID review. |

## 7. Production Monitoring

**Tracing tool: a single structured run-log line per night, written to the GitHub Actions console.**
NOT Arize Phoenix, Langfuse, LangSmith, or any SaaS/Python observability stack — those are explicitly
out of scope for a one-call-per-night internal tool with no always-on server. The Actions run history
IS the trace store; one greppable JSON-ish line per run makes fallback-rate and cost visible without
infrastructure. (The full run-logging mechanism is Phase 6 / REL-02; Phase 5's monitoring is
deliberately minimal — it just emits the LLM-specific fields into that one line.)

**Key Metrics to Track** (the fields on the nightly run-log line):
- `renderPath` — `"llm"` or `"template"` (the single most important signal — surfaces fallback rate).
- `fallbackReason` — failure class when `renderPath === "template"` (zod / timeout / max_tokens / refusal / transport / none). Never the key or webhook URL.
- `model` — the pinned model ID actually used (catches a silent model swap).
- `inputTokens` / `outputTokens` / `estCostUsd` — from the SDK response `usage`; the cost-drift signal.
- `latencyMs` — wall-clock of the call (catches creep toward the 20 s ceiling).

**Alert Thresholds** (lightweight — this is an internal nudge tool, no pager):
- **Any night the workflow run fails** (non-zero exit) → GitHub Actions' built-in failure notification (REL-02). This is the only true "alert".
- **The card posts with a degraded note** is itself the human-visible signal — the team sees the LLM prose is missing.
- **Soft watch (no automation):** if `renderPath === "template"` for several consecutive weekdays with the same `fallbackReason`, the LLM path needs attention — caught by glancing at the run history, not by a paging rule.
- **Cost:** no threshold needed — bounded by `max_tokens` + the sub-cent/night budget (§4b); the `estCostUsd` field is a sanity eyeball, not an alarm.

**Smart Sampling Strategy:**
Volume is ~22 runs/month, so sampling is "review the interesting ones," not statistical sampling.
Weight human review toward signal: any night with `renderPath === "template"` (why did the LLM
fail?), any card a PM flags as an unfair/missed meeting flag (feed back into `labelled-events.json`),
and any header sentence that reads off-brand (feed the creative lead's voice review). A clean,
all-sorted night on the LLM path needs no review. This is the signal-metric-divergence early-warning
loop kept appropriately tiny for the tool's scale.
