# Phase 3 — UI Design Contract

> Visual and interaction contract for the **Google Chat Cards v2 message**. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **This is NOT a web app.** The "UI" is a single Cards v2 JSON message delivered via incoming webhook. There is no DOM, CSS, shadcn, component library, px spacing scale, or responsive breakpoint. Google controls all rendering (Roboto, fixed paddings, fixed widget styles). This contract therefore replaces the standard web sections with their **Cards v2 equivalents**: widget hierarchy, the supported inline-HTML subset, emoji status markers, and inline `<font color>` values.
>
> The design is **already locked** — decided by the brand designer over ~15 visual iteration rounds (CONTEXT.md D-01..D-25). The canonical visual reference is `design/chat-card-mockups.html` (round 15). This document **formalises** those decisions into verifiable rules; it does not re-decide them. Anything genuinely unspecified is marked **Claude's discretion (planner)**, not asked.

## Design System

| Property | Value |
|----------|-------|
| Tool | **Not applicable** — Google Chat Cards v2 (Google-controlled rendering, no custom CSS, no design-system tooling) |
| Preset | not applicable |
| Component library | **Cards v2 widgets only**: `cardHeader`, `section`, `decoratedText`, `textParagraph`, `divider`, `buttonList` + `openLink` |
| Icon library | **Unicode emoji** for status markers (D-10) + one hosted **PNG** avatar (D-04/D-07). No SVG icon set (SVG support in Cards v2 is unreliable, D-04) |
| Font | **Roboto** (Google-imposed; no custom fonts in Cards v2, D-02). Brand typefaces (Host Grotesk / Oldschool Grotesk Compressed) are NOT used in the card. Brand expression here = the avatar, the voice/copy, and clean structure (D-05) |

**Why the brand type is absent:** Cards v2 has no custom-font support (verified, RESEARCH.md / developers.google.com/workspace/chat/format-messages). This is accepted (D-02). Do not attempt `<font face>` — it is ignored or breaks rendering.

## Widget Hierarchy (replaces "Layout")

The card is a single `cardsV2[0].card` with `cardId: "studio-checkin"`. Top-to-bottom widget order is **contractual** (MSG-02 mandates verdict → week rollup → rows → flags; the locked design places the week bar in the footer and groups flags *inside* each designer row):

| Order | Widget | Content | Source |
|-------|--------|---------|--------|
| 1 | `cardHeader` | `imageType: CIRCLE` avatar PNG + `title: "Solvd Studio Check-in"` + `subtitle: "Tomorrow · {Weekday Date}"` (e.g. "Tomorrow · Thursday 4 June") | D-06, D-07 |
| 2 | `section` → `textParagraph` | **Verdict lead** — one nameless line, `<b>` (see Copywriting Contract). On clean nights add the status line. | D-08, D-12, D-13 |
| 3 | `section` → N × `decoratedText` (busy nights only) | **Per-designer rows**, separated by `divider`. Each row carries its own nested ⚠️ tentative and 📄 brief lines inside the main `text`. | D-08, D-09, D-14, D-16 |
| 4 | `section` → `buttonList` | Single **"Open in Productive"** `openLink` button → tomorrow's scheduling view. | D-24 |
| 5 | `section` (with `header`) → 2 × `textParagraph` | **Footer week bar.** Section header `"Remaining studio time this week"`; line 1 = dot bar; line 2 = caption `"{X}h booked · {Y}h open"`. | D-23 |

**Clean-night collapse (D-17 / MSG-05):** when no designer is underbooked/overbooked AND there are no brief flags, **omit widget group 3 entirely** — render only header + verdict + status line + button + week bar. Length scales with severity.

### Per-designer row anatomy (D-09 — the load-bearing rule)

Each row is **one `decoratedText`** widget. ALL text lives in the main `text` field, lines separated by `<br>`:

- **Line 1:** `{status emoji} <b>{Name}</b> — {coloured status}` — name bold, status in the inline `<font color>` for its state.
- **Line 2:** the booked/open detail, greyed: `<font color="#5f6368">{Nothing booked | Xh booked | …}</font>`
- **Nested flag lines (0–n):** each on its own `<br>` line, body size — the ⚠️ tentative line and/or 📄 brief line(s).

**Hard rule (D-09):** detail and flag lines go in the main `text` (body size), **NEVER** in `topLabel`/`bottomLabel`. Those label fields render in a fixed, unresizable small-caption font (`topLabel` truncates, `bottomLabel` wraps) — using them violates the hierarchy-by-colour/weight-not-size principle. Hierarchy is expressed by **colour** (`<font color>`) and **weight** (`<b>`), never by size.

**Status emoji:** render **inline at the start of `text`** (the literal emoji character), matching the mockup, rather than `startIcon.iconUrl`. D-10 specifies emoji, not icons. (`startIcon` PNG dots are a valid alternative but not the default — planner's discretion.)

## Spacing Scale

**Not applicable — Google Chat Cards v2.** There is no px spacing scale and no `style`/`class` control. Google fixes all widget paddings, gaps, and the avatar size. The only spacing levers available to the renderer are:

| Lever | Use |
|-------|-----|
| `<br>` | Line breaks **within** a `decoratedText.text` (the only way to stack the detail/flag lines, D-09) |
| `divider` widget | Visual separation **between** designer rows (per mockup `<hr>`) |
| `section` boundary | Grouping: verdict / rows / button / week bar are separate sections; the footer section uses a `header` to title the week bar |

Do not attempt to inject blank lines, &nbsp; padding, or any pixel spacing. Exceptions: none.

## Typography

**Roboto only (Google-imposed).** The renderer cannot set size or family; it controls **weight and colour** via the inline-HTML subset. Cards v2 supported tags: `<b> <i> <u> <s> <font color> <a> <br>` (and `<code>/<pre>` for monospace). No `<font face>`, no `<font size>`, no background/highlight (D-02, D-03).

| Role | How it is expressed | Rule |
|------|---------------------|------|
| Header title | `cardHeader.title` (Google styles it) | "Solvd Studio Check-in" — fixed |
| Header subtitle | `cardHeader.subtitle` (Google styles it) | "Tomorrow · {Weekday Date}" |
| Verdict lead | `<b>…</b>` in a `textParagraph` | Bold; sentence case; nameless (D-12) |
| Designer name | `<b>{Name}</b>` on row line 1 | Bold = the emphasis weight |
| Status (open/full/over) | `<font color>` on row line 1 | Colour carries state, not size |
| Detail / flag lines | plain or `<font color="#5f6368">` | Body weight, greyed for hierarchy |
| Week bar | `textParagraph`; dots may be wrapped in `<code>…</code>` | See Color + the dot-bar note |

**Monospace caveat (D-23):** Roboto Mono is **not guaranteed** to render in card text; the only reliable monospace is `<code>/<pre>`. The `●`/`○` dot glyphs are near-fixed-width and render acceptably in proportional Roboto, but if alignment drifts in the real client, wrap the dot run in `<code>…</code>`. **Verify visually in the real Chat space during execution** (RESEARCH.md A1).

## Color

**Not applicable as a 60/30/10 web split.** Cards v2 forbids background/highlight colour (D-03); the card surface is Google's white. Colour exists **only** via inline `<font color>` and emoji. The brand yellow appears in **exactly one** native place: the avatar PNG (D-03/D-05).

### Surface (Google-controlled, not settable)

| Role | Value | Note |
|------|-------|------|
| Card surface | Google white (~`#ffffff`) | Not settable; no background colour allowed (D-03) |
| Brand yellow `#FEFD5C` | **Avatar PNG only** | Never as card text/fill — yellow-on-white is 1.08:1, fails WCAG (banned, D-03 + visual-rules.md) |

### Inline `<font color>` palette (the renderer's entire colour vocabulary — D-11, D-23)

| Token | Hex | Reserved exclusively for |
|-------|-----|--------------------------|
| Status red | `#d93025` | Open time ("Xh open") **and** the degraded/"couldn't reach" 🤖 state |
| Status green | `#188038` | Full day ("full day") |
| Overbooked amber-brown | `#b06000` | Overbooked only ("Xh over") |
| Muted grey | `#5f6368` | All greyed detail/flag text (booked/open detail, client names, job codes, captions) |
| Open-dots grey | `#c9ccd1` | The **empty** portion of the week-bar dot run only |

Booked-dots / "ink" in the bar use the **default Roboto text colour** (no `<font>` needed) — the dark glyph reads as "ink, booked".

**No other colours.** No secondary accent beyond the avatar yellow (brand rule: yellow is the only accent, and it is dark-surface-only). Destructive colour: **none** — this surface has no destructive actions (it is a read-only, one-way nudge).

| ⚪ | On leave |
| 🤖 | Couldn't read this designer / couldn't reach source |

## Accessibility / Contrast Contract

- **Yellow-on-white is banned** (D-03). `#FEFD5C` on white = **1.08:1** — fails every WCAG level (visual-rules.md). Yellow is permitted **only** baked into the avatar PNG (white asterisk on a black circle, D-07), never as card text or any fill.
- **State is never colour-alone.** Every status is conveyed by **emoji + colour + weight** together (🔴 + red + position), so colour-blind readers and any client that flattens `<font color>` still parse the state from the emoji and the words. This is a hard requirement — do not introduce any status that relies on colour only.
- **Inline status colours pass on white:** red `#d93025` (~5.0:1), green `#188038` (~4.9:1), amber-brown `#b06000` (~5.3:1), muted grey `#5f6368` (~5.9:1) — all clear the 4.5:1 normal-text floor. Open-dots grey `#c9ccd1` is decorative (the empty fuel-gauge segment), not text, so its low contrast is acceptable.
- **Avatar must be anonymously fetchable** by Google's server-side image renderer (public HTTPS PNG; verify in incognito, RESEARCH.md Pitfall 4). If unreachable the card still posts (header just lacks the image) — degraded brand fidelity, not a failure.
- **HTML-escape all dynamic text** (client names, job codes) before inserting into `text` (`&`→`&amp;`, `<`→`&lt;`, `>`→`&gt;`) so a stray `&`/`<` can't break one row's rendering (RESEARCH.md Pitfall 3 / V5).

## Copywriting Contract

Voice: Solvd tone-of-voice — direct, confident, human, **Australian English**, **sentence case**, no jargon, no hollow superlatives (tone-of-voice.md). The verdict **never names a person** (D-12) — names appear only in designer rows (MSG-03, collective-nudge ethos; never a PM).

### Verdict lead — nameless, scenario-adaptive (D-12, D-13)

| Scenario | Verdict line (locked) |
|----------|----------------------|
| Clean | **"All sorted for tomorrow."** + status **"Three designers fully booked. Nothing to action."** (status line locked verbatim) |
| Two open | **"Two designers have open time tomorrow."** |
| One open | **"One designer has a bit of open time tomorrow."** |
| All open | **"The whole studio's light tomorrow."** |
| Overbooked (one) | **"One designer's a bit over tomorrow."** |
| All over | **"The whole studio's overbooked tomorrow."** |
| Mixed | **"A couple of things to sort tomorrow."** |
| Briefs only | **"Everyone's booked — but {N} brief(s) need finishing."** |
| On leave | Leave **never** drives the verdict. Even with 2 of 3 on leave, if the rest is fine the verdict is **"All sorted for tomorrow."** (D-13 final bullet) |
| Could read only one | **"I could only read one designer tonight."** (nameless; D-19) |

### Per-designer row copy

| Element | Locked format | Source |
|---------|---------------|--------|
| Open status | `🔴 <b>{Name}</b> — <font color="#d93025">{X.X}h open</font>` | D-09, D-11 |
| Full status | `🟢 <b>{Name}</b> — <font color="#188038">full day</font>` | D-09, D-11 |
| Overbooked status | `🟠 <b>{Name}</b> — <font color="#b06000">{X.X}h over</font>` | D-09, D-11 |
| Booked detail | greyed `Nothing booked` / `{X.X}h booked` / `{X.X}h booked` under the name | mockup |
| **Tentative** (⚠️) | **`⚠️ {X.X}h tentative (on top) · {Client}`** — client + hours, **NO job code** (allocations carry no code). `(on top)` is **mandatory and locked** (signals additive). | D-14, D-15 |
| **Brief** (📄) | **`📄 {label} · {CODE} · {X}h`** — labels are exactly **`No brief`** / **`Brief empty`** / **`Not briefed`**; always include job code + booked hours. Nested under the booked designer, NOT a separate section. | D-16 |

**Tentative is never counted** toward booked/open/over — those figures are confirmed-only. The ⚠️ "(on top)" line must make the additive nature visually unambiguous (D-15, MSG-07).

### Week-bar footer (D-23)

| Element | Locked copy |
|---------|-------------|
| Section header | **"Remaining studio time this week"** |
| Dot bar | `●` filled = booked (default ink), `○`/greyed `●` = open (`#c9ccd1`); fuel-gauge — full bar = booked up, mostly empty = lots to fill |
| Caption | **"{X}h booked · {Y}h open"** (both figures, greyed) |

Dot count: **Claude's discretion (planner)** — RESEARCH.md recommends fixed 10 dots, `filled = round(bookedMin / totalMin * 10)` computed from `rollup` minutes, rendered display-only (the renderer formats, never computes new figures). Confirm the count rule in planning; it is a presentation detail, not locked verbatim.

### State / error variants (always posts — REL-01, MSG-04)

| Variant | Locked copy | Source |
|---------|-------------|--------|
| **Degraded** (source unreachable, e.g. Productive) | **"🤖 Couldn't reach Productive tonight."** + **"No booking figures this run. I'll have them tomorrow evening — worth a check in Productive yourself in the meantime."** Names what it couldn't reach; still posts. | D-18, REL-01 |
| **Per-designer miss** (one/two unreadable) | **🤖** row next to that person; never fake an empty/zero. Verdict nameless (e.g. "I could only read one designer tonight."). | D-19 |
| **Holiday tomorrow** | **"🎉 Public holiday tomorrow — {name}. No check-in needed. Enjoy…"** Own short warm message; no rows/bar. | D-20 |
| **Studio closure / offsite** | **"📦 Studio's out tomorrow — team offsite. No check-in needed. Back {day}."** Own message; no rows/bar. | D-21 |
| **On-leave row** | Minimal — only **"on leave / Full day off."** Nothing more. | D-22 |
| **Half-day / partial leave** | Treated as a normal availability row for the hours present, with note **"On leave until midday · {X}h booked"**. (Exact "until midday" wording needs absence detail passed via `RenderContext` — see Open Items.) | D-22 |

**Empty state:** there is no conventional empty state — the card **always** posts (MSG-04). The "nothing to do" case is the **clean-night** verdict above, which is positive, not empty.

### Primary CTA (D-24)

| Element | Locked value |
|---------|--------------|
| Button label | **"Open in Productive"** (single `openLink` button; sentence case; verb + noun) |
| Target | Tomorrow's scheduling view: `https://app.productive.io/34092-solvd-agency/scheduling/bookings?date={TARGET_YYYY-MM-DD}&filter=NzQ5NTY2&groupBy=people` (`NzQ5NTY2` = base64 of `749566`, the design-team filter; `date` = `report.targetDay`). Verify the URL/filter resolves during planning. |

**No destructive actions.** This surface is a one-way, read-only nudge — there are no destructive controls and no confirmation flows. (Total-failure alerting relies on GitHub Actions' built-in failed-run email, D-25 — not a card element.)

## Open Items for Planner (genuinely unspecified — not asked)

1. **Dot-bar count rule** (D-23): fixed-10 proportional vs dots-per-hour. Recommend fixed 10, `round(bookedMin/totalMin*10)`, display-only from `rollup`.
2. **Half-day leave wording source** (D-22): "until midday" needs absence end-time, which `StudioReport` doesn't carry today. Pass via `RenderContext` (avoids touching `domain`), or fall back to a simpler "Part-day leave · {X}h available". Planner decides.
3. **Emoji vs hosted PNG dots** for the status gutter: emoji-inline is the recommended default (D-10 says emoji); `startIcon` PNG is the alternative.
4. **`<code>` wrapping the dot bar**: decide after a visual check in the real Chat space (alignment).
