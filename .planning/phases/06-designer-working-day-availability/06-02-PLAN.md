<objective>
Fetch each monitored designer's real working-day pattern from Productive (`person.availabilities.working_hours`), parse it at the zod boundary, map it to exact per-weekday rostered minutes, and expose a `rosteredMinutes(designerId, dateKey)` lookup on `GatherResult` that satisfies the Plan-01 `StudioReportInput.rosteredMinutes` contract. A designer whose availability can't be read is omitted from `assessedDesigners` (D-06) so the report names them "couldn't read" instead of inventing open time.

Purpose: This is the data half of CAP-06 — the live source of truth (Anisha off Wed & Fri, etc.) that replaces the flat-450 assumption. Live-confirmed shapes (06-CONTEXT specifics) drive the fixtures.

Output: A new availabilities schema + raw type; an availability mapper returning per-weekday minutes; a `/people` pull in gather with per-designer degrade; the `rosteredMinutes` lookup on GatherResult; mapper + gather tests pinning the live shapes.

Open decision D-01 RESOLVED (Claude's discretion): availabilities are pulled via a DEDICATED `/people?filter[id]=...` call, NOT sideloaded on the existing `/bookings` include. Rationale: the `/bookings` include chain already broke once when relationships weren't included (02-04 SUMMARY, gather.ts lines 401-413) and is brittle; `availabilities` is a person ATTRIBUTE (not a relationship), so a clean person-scoped pull mirrors the `personFilter` pattern (gather.ts line 411), keeps the proven bookings query untouched, and makes per-designer degrade (D-06) straightforward.
</objective>
