    - Window enumeration via restOfWeekWindow already tested in 01-01; here assert the rollup that sits on top:
    - Rollup totals (net of time-off, D-09): for a Tue 2026-06-09 target with window Tue–Fri (4 days), 3 designers, NO absences and NO bookings → totalMin = 4 days × 3 designers × 450 = 5400 (totalHours 90.0); openMin = 5400 (everyone fully open); openHours 90.0.
    - One designer fully off (absence 450) on one window day reduces totalMin by 450 (net of time-off, D-09): same scenario but designer A off Wednesday → totalMin = 5400 − 450 = 4950 (totalHours 82.5).
    - Holiday in window (D-10): same Tue target but holidays = {Thu 2026-06-11} → window is Tue/Wed/Fri (3 working days), totalMin = 3 × 3 × 450 = 4050; the holiday contributes 0 and is not counted as a working day.
    - Friday rollover (D-08): a Friday-evening run whose target is Mon 2026-06-08 → window Mon–Fri (5 days), totalMin = 5 × 3 × 450 = 6750 with no absences.
    - Missing designer (D-18): input contains data for only A and B (C absent from all input arrays) → report.missingDesigners deep-equals ["C"] (or contains C); designers array still reports A and B; the function does NOT throw.
    - Empty input (D-19): bookings = [], absences = [] for the full roster → every designer underbooked with full available open (450 each on the target day, per 01-02 D-17); missingDesigners = [] (roster all present-but-empty is different from missing — document the distinction: present-with-no-bookings vs absent-from-input).
    - Determinism: calling computeStudioReport twice with identical inputs returns deep-equal output (no randomness, no I/O).
    - Garbage tolerance (D-19): a booking with NaN minutes or an unexpected extra field does not throw — minutes coerced to 0, report still produced.
